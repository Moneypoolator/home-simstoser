
# Анализ состояния проекта S3-хранилища

Проект представляет собой S3-совместимый сервер хранения файлов на C++20 с веб-интерфейсом. Архитектура модульная, используются современные библиотеки (Boost.Asio/Beast, OpenSSL, nlohmann/json, glog). Реализована аутентификация AWS Signature V4, ролевая авторизация, многопоточный асинхронный сервер, поддержка HTTPS, ограничение частоты запросов, multipart и stream upload.

## Достоинства

1. **Архитектура** – чёткое разделение на компоненты: `authenticator`, `authorizer`, `file_manager`, `request_handler`, `server`, `rate_limiter`. Это упрощает поддержку и тестирование.

2. **S3-совместимость** – реализована аутентификация AWS Signature V4, базовые операции (PUT, GET, DELETE, LIST), multipart uploads. Можно использовать стандартные S3-клиенты.

3. **Безопасность** – защита от path traversal (`is_path_safe`), поддержка HTTPS с самоподписанными сертификатами и Let’s Encrypt, rate limiting и DDoS-защита.

4. **Производительность** – асинхронный I/O (Boost.Asio), пул потоков, неблокирующий acceptor.

5. **Функциональность** – ролевая авторизация (ADMIN, MANAGER, CONTRIBUTOR, VIEWER, GUEST), ACL для ресурсов, загрузка по частям, стриминговая загрузка, прогресс операций.

6. **Документация** – подробный README с инструкциями по сборке, запуску, API, roadmap.

## Недостатки и проблемы

### Критические (требуют немедленного исправления)

1. **SSL-ветка не использует аутентификацию и авторизацию**
В `server::handle_ssl_session` создаётся `request_handler` без передачи `_authenticator` и `_authorizer`, а также не вызываются `set_auth_enabled` / `set_authorization_enabled`. HTTPS-запросы работают без проверки подписи и прав доступа.

2. **Нет проверки временной метки в аутентификации**
`authenticator::is_timestamp_valid` всегда возвращает `true`. Это позволяет replay-атаки (запросы можно повторять бесконечно).

3. **Отсутствует канонический query string при подписи**
В `generate_signature` используется пустая строка вместо реальных query-параметров. Подпись для запросов с параметрами (например, `?list-type=2`) будет неверной.

4. **Чтение всего файла в память**
`file_manager::download_file` и `upload_file` загружают файл целиком в `std::vector<char>`. При больших файлах (>RAM) сервер упадёт.

5. **Объединение частей multipart upload в память**
`merge_parts` читает каждую часть в `std::vector<char>`. Для большого числа частей или больших частей – катастрофическое потребление памяти.

6. **Path traversal можно обойти через симлинки**
`is_path_safe` использует `weakly_canonical`, но не проверяет, не ведёт ли симлинк за пределы storage_dir. Злоумышленник может создать символическую ссылку на `/etc/passwd`.

7. **Не загружаются пользователи из users_file**
В конструкторе `s3_server` есть `TODO: Загрузка пользователей из файла`. Авторизация работает, но все пользователи теряются после перезапуска.

### Архитектурные и дизайнерские

8. **Дублирование кода и пропущенные вызовы**

- В `handle_ssl_session` не используется `rate_limiter`.

- Нет периодического вызова `rate_limiter::cleanup` – память для IP-адресов будет расти.

- `check_policy_permissions` всегда возвращает `false` – политики доступа не работают.

- Групповые права в ACL не реализованы (`TODO`).

9. **Смешение уровней абстракции**
`request_handler` напрямую работает с `std::vector<char>` и вызывает методы `file_manager`, принимающие векторы. Нет потоковой обработки.

10. **Закрытие соединения после каждого запроса**
Нет поддержки keep-alive. Для каждого запроса создаётся новый сокет и поток. Снижает производительность при частых мелких запросах.

11. **Отсутствие кэша метаданных**
`list_files` каждый раз рекурсивно обходит всю файловую систему – медленно для тысяч файлов.

12. **Нет graceful shutdown**
При остановке сервера текущие запросы обрываются, временные файлы могут не очиститься.

### Безопасность

13. **Потенциальная инъекция команд**
`std::system` для вызова `openssl` в `main.cpp` – хотя аргументы экранированы, лучше использовать OpenSSL API.

14. **Отсутствие лимитов**
Нет ограничений на размер загружаемого файла, количество частей, объём временных файлов. Злоумышленник может заполнить диск.

15. **Проверка клиентских сертификатов отключена**
`verify_client = false` – сервер не проверяет клиентские сертификаты даже при включённом SSL.

### Качество кода

16. **Отладочный код**
`getenv("DEBUG_AUTH")` в `authenticator.cpp` – лучше использовать флаг логирования.

17. **Голые указатели**
`request_handler` хранит `authenticator*` и `authorizer*` без проверки на `nullptr`. При `_auth_enabled=true` и `_authenticator=nullptr` – крах.

18. **Магические числа**
Лимит тела запроса 100 MB жёстко задан в `server.cpp`.

19. **Дублирование функций**
`hex_encode`, `generate_id` реализованы в нескольких местах.

20. **Тесты отсутствуют**
Хотя `GTest` подключён, нет ни одного юнит-теста.

## Направления развития и улучшения

### Ближайшие (критические исправления)

1. [X] **Исправить SSL-ветку** – передать `_authenticator`, `_authorizer`, `_rate_limiter` в request_handler для HTTPS.

2. [ ] **Реализовать проверку timestamp** – допустимое отклонение 15 минут, парсинг `x-amz-date`.

3. [ ] **Добавить canonical query string** – собрать отсортированные параметры, закодировать, включить в подпись.

4. [ ] **Переделать file_manager на потоковую обработку** – использовать `std::ifstream` с буфером, `sendfile` для отправки, копирование файлов для объединения частей.

5. [ ] **Усилить `is_path_safe`** – после `canonical` проверить, что результирующий путь начинается с `storage_dir`.

6. [ ] **Загружать пользователей из файла** – реализовать парсинг CSV/JSON, обновлять `_users`.

7. [ ] **Добавить лимиты** – максимальный размер файла, количество частей, общий объём временных файлов.

### Среднесрочные (улучшение надёжности и производительности)

8. **Кэш метаданных** – `unordered_map` с инвалидацией при записи/удалении, периодическое обновление.

9. **Keep-alive соединения** – читать несколько запросов на одном сокете, таймаут.

10. **Периодическая очистка** – вызывать `rate_limiter::cleanup` и `file_manager::cleanup_old_uploads` раз в минуту.

11. **Потоковое объединение частей** – использовать `std::ofstream` и копировать части через буфер фиксированного размера (например, 1 MB).

12. **Graceful shutdown** – дождаться завершения активных запросов, закрыть acceptor, остановить io_context, удалить временные файлы.

13. **Вынести конфигурацию** – единый файл (JSON/YAML) с возможностью перезагрузки без остановки.

14. **Реализовать групповые права и политики** – дополнить `check_policy_permissions`, загружать политики из файла.

### Долгосрочные (функциональность и экосистема)

15. **Поддержка бакетов** – изолированные пространства имён, метаданные бакетов.

16. **Версионирование объектов** – хранить старые версии, настраиваемые правила жизненного цикла.

17. **Range-запросы** – `Range: bytes=0-1023` для частичной загрузки.

18. **Метаданные объектов** – сохранение и возврат заголовков `x-amz-meta-*`.

19. **Prometheus метрики** – эндпоинт `/metrics` с гистограммами задержек, счётчиками ошибок.

20. **Docker-образ** – `Dockerfile` и `docker-compose` для простого развёртывания.

21. **WebDAV** – альтернативный протокол для совместимости с клиентами WebDAV.

22. **Шифрование на диске** – прозрачное шифрование (AES-256) с управлением ключами.

23. **Интеграционные тесты** – на базе `GTest` и `curl` для проверки S3 API.

### Улучшение качества кода

24. **Заменить голые указатели на `std::shared_ptr` или ссылки** – гарантировать время жизни.

25. **Добавить `nodiscard`** – для функций, возвращающих bool успеха.

26. **Перейти на `std::format` (C++20)** – вместо `stringstream` для производительности.

27. **Вынести парсинг Authorization header в отдельный класс** – покрыть тестами.

28. **Написать unit-тесты** – для аутентификации, авторизации, rate limiter, проверки путей.

29. **Использовать `spdlog` вместо `glog`** – более современный, асинхронный, с поддержкой ротации.

## Заключение

Для production-использования необходимо усилить безопасность (проверка времени, защита от симлинков, лимиты), переделать работу с большими файлами на потоковую, добавить кэширование и поддержку keep-alive. Реализация AWS Signature V4 требует доработки (query string, timestamp). Архитектурные ошибки (SSL-ветка) должны быть исправлены в первую очередь.
